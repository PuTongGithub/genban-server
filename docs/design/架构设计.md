# Genban 架构设计

## 架构图

![Genban 架构图](Genban系统架构图.drawio.png)

## 技术栈

- **编程语言**: Python
- **Web 框架**: FastAPI
- **数据库**: SQLite

## 架构概述

Genban 采用分层架构设计，从外到内分为以下层次：

1. **Gateway 层**: 外部对接管理，统一对外提供接口
2. **User 层**: 用户管理模块，负责用户管控接入
3. **Agent 层**: AI 调用流程编排，负责单次 AI 调用的完整流程
4. **Modules 层**: 应用功能模块化，为 Agent 提供个性化能力
5. **Assistant 层**: 助手核心实现，利用 Agent 实现流程控制，并向 Agent 注入Modules 能力
6. **Storage 层**: 数据存储管理，负责数据的持久化与访问
7. **Provider 层**: 对外请求管理，负责调用外部服务接口

---

## 模块划分

### 1. Gateway（外部对接管理）

**职责**: 统一对外接口管理，负责外部请求的路由、认证与响应

**功能**:
- **API 路由**: 统一管理所有对外 API 接口
- **请求认证**: JWT Token 验证、权限校验
- **日志记录**: 请求/响应日志

---

### 2. User（用户管理模块）

**职责**: 用户身份认证与权限管理，为 Assistant 提供用户管控接入

**功能**:
- **用户管理**: 管理用户账户
- **身份认证**: JWT Token 生成与验证

---

### 3. Assistant（助手核心实现）

**职责**: AI 个人助理的核心实现，整合 Agent 流程控制与 Modules 工具模块化能力

**关键设计**:
- 提供 HTTP 接口供前端交互（`/submit`、`/stream`）
- 内部使用 AgentWorkerManager 管理消息队列和 Worker
- 每个用户的 UserAgentWorker 复用 Agent 实例处理消息
- 产生的新消息通过 SSE 推送到前端

**数据流**:
1. 前端调用 `/submit` 提交消息 → 创建 Chat 并入队
2. AgentWorkerManager 从队列取出消息 → 分发给对应 UserAgentWorker
3. UserAgentWorker 调用 Agent 处理 → 产生的新消息推给 NewChatHook → NewChatHook 下发消息给 StreamManeger
4. `/stream`接口订阅 StreamManeger → 前端得到 SSE 推送

---

### 4. Modules（应用功能模块化）

**职责**: 应用功能的模块化，为 Agent 提供个性化的应用能力

**子模块**:

| 名称                 | 职责                                    |
| ------------------ | ------------------------------------- |
| FileSystem（文件系统模块） | 向 Agent 提供文件系统操作能力                    |
| Bash（命令行模块）        | 向 Agent 提供命令行调用能力                     |
| Setting（配置模块）      | 系统配置与用户个性化设置管理                        |
| Memory（记忆模块）       | 实现长期记忆管理机制，后续将接管对话管理功能                |
| Task（任务模块）         | 向 Agent 提供管理 SubAgent 的能力，用来完成用户布置的任务 |
| Cron（定时模块）         | 向 Agent 提供定时唤醒                        |
| Schedule（日程管理）     | 管理用户日程                                |
| Skills（技能模块）       | 向 Agent 提供可扩展的 Skill 能力，Skill 文件通过 FileSystem 模块管理 |

---

### 5. Agent（AI 调用流程编排）

**职责**: Agent 是单次 AI 调用的流程编排器，负责整合模型调用、工具调用和生命周期钩子，完成从用户输入到 AI 响应的完整流程。

**核心能力**:
- **单次调用编排**: 接收用户输入，通过钩子链处理，调用大模型，处理工具调用循环，返回所有新增的 Chat
- **流式响应支持**: 通过流式调用获取模型响应，实时触发钩子
- **工具调用循环**: 自动处理模型触发的工具调用，支持多轮工具调用（最大50次迭代）
- **生命周期钩子**: 在关键节点提供 7 个钩子点位，支持自定义逻辑注入

**子模块**:

| 名称 | 职责 | 关键组件 |
|------|------|----------|
| Model（模型） | 管理大模型接口的统一调用 | ModelCaller、ModelProvider |
| Chats（对话） | 定义 Agent 对话内容格式 | Chat、Message、ChatType、chat_factory |
| Tools（工具） | 实现 Agent 工具调用能力 | BaseTool、ToolCaller、ToolRegistry |
| Hooks（钩子） | 支持 Agent 生命周期的回调与事件监听 | BaseHook、HookManager、HookRegistry |

**钩子点位（按执行顺序）**:

1. **ModelHook**: 决定使用哪个模型（首个执行）
2. **ChatHook**: 处理输入对话
3. **PromptHook**: 处理提示词对话
4. **ChatsHook**: 处理历史对话列表
5. **ToolsHook**: 处理工具注册表
6. **NewChatHook**: 处理新增的 Chat（异步执行，包括模型响应和工具结果）
7. **CompleteHook**: 处理完成时的所有新增 Chat 列表（异步执行）

**使用方式**:

```python
from src.agent.agent import Agent
from src.agent.chat_factory import chat_factory

# 创建 Agent 实例
agent = Agent(
    user_id="user001",
    tools=[WeatherTool(), CalculatorTool()],  # 可选：初始工具
    hooks=[LoggingHook(), MetricsHook()]       # 可选：初始钩子
)

# 或动态注册工具和钩子
agent.register_tool(WeatherTool())
agent.register_hook(LoggingHook())

# 创建用户输入 Chat
input_chat = chat_factory.create_user_chat(user_id="user001", user_input="今天天气如何？")

# 执行 Agent 调用，返回本次所有新增的 Chat 列表
new_chats = agent.run(chat=input_chat)
# new_chats 包含：assistant 响应 Chat、tool 调用结果 Chat 等
```

**模块边界**:
- **输入**: 单个 Chat 对象（用户输入）
- **输出**: 本次调用新增的 Chat 列表（assistant 响应、tool 结果等）
- **不处理**: 历史对话的存储与检索（由上层调用方管理）、多轮对话的会话管理
- **依赖**: ModelCaller（模型调用）、ToolCaller（工具执行）、HookManager（钩子管理）

**关键设计决策**:

1. **Agent 复用设计**: 每个用户的 UserAgentWorker 持有一个 Agent 实例，在处理多个消息时复用，避免重复初始化开销

2. **单次调用原则**: Agent 只负责单次 AI 调用的完整流程，不维护对话状态，状态管理由调用方负责

---

### 6. Storage（数据存储管理）

**职责**: 负责数据的持久化存储与访问，为上层模块提供统一的数据操作接口

**子模块**:
- **Database**: 数据库连接与管理（SQLite）
- **Vector Store**: 向量数据库存储（用于记忆语义检索）
- **File Store**: 文件存储管理（本地文件系统、对象存储等）

---

### 7. Provider（对外请求管理）

**职责**: 统一管理所有对外部服务的调用，封装第三方接口的请求逻辑

**子模块**:
- **API DashScope**: 阿里云 DashScope 大模型 API 封装
- **API Zai SDK**: 智谱 AI SDK 封装（网络搜索等）

---

## 日志规范

### 日志架构

项目采用统一的日志管理架构，所有模块通过 `src/common/logger.py` 获取日志记录器

### 各层日志职责

| 层次 | 日志级别 | 记录内容 |
|------|----------|----------|
| **Gateway 层** | INFO/ERROR | HTTP 请求接收、响应、异常 |
| **Assistant 层** | INFO/DEBUG | 消息队列处理、Worker 状态 |
| **Agent 层** | DEBUG/INFO | 执行流程、模型调用、工具调用 |
| **Modules 层** | DEBUG/INFO | 文件操作、命令执行、Skill 操作 |
| **Provider 层** | INFO/ERROR | 外部 API 调用、响应、异常 |
| **Storage 层** | DEBUG/ERROR | 数据库操作、文件存储 |

### 日志级别使用规范

| 级别 | 使用场景 |
|------|----------|
| **DEBUG** | 详细的调试信息，如函数入参、执行流程步骤、循环迭代次数 |
| **INFO** | 重要的业务操作，如请求接收、处理完成、外部 API 调用 |
| **WARNING** | 警告信息，如权限拒绝、资源不存在、非预期的输入 |
| **ERROR** | 错误信息，如异常捕获、操作失败、外部服务错误 |
| **CRITICAL** | 严重错误，系统级错误、关键组件故障 |

### 日志配置

日志配置文件 `src/config/jsons/logging_config.json`：

```json
{
  "level": "INFO",
  "format": "%(asctime)s - %(name)s - %(levelname)s - %(message)s",
  "file_enabled": true,
  "log_dir": "logs",
  "log_file": "app.log",
  "rotation_when": "midnight",
  "backup_count": 7
}
```
