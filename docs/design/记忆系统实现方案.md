# Genban 记忆系统实现方案

本文档描述 Genban 长期陪伴型个人 Agent 的记忆系统设计与实现流程。该设计基于对 PlugMem（ICML 2026）论文与源码的分析，针对"任务执行 + 长期陪伴"的双重场景进行了适配。

---

## 一、记忆图结构

记忆系统采用**异构图**结构存储知识，包含五种节点和两条并行的知识树。

### 1.1 五种节点

| 节点类型       | 节点名称           | 含义                       | 关键字段                                | 是否参与向量检索               |
| :--------- | :------------- | :----------------------- | :---------------------------------- | :--------------------- |
| **互动记录节点** | EpisodicNode   | 单步交互的原始痕迹                | 观察内容、执行动作、子目标、状态、奖励、时间戳、会话ID、原始观察向量 | 是（原始 observation 保真检索） |
| **事实节点**   | SemanticNode   | 从交互中提取的事实知识（用户事实 + 领域知识） | 事实文本、向量嵌入、标签列表、来源、可信度、是否活跃、事实来源类型   | 是                      |
| **标签节点**   | TagNode        | 对事实进行分类的语义标签             | 标签文本、向量嵌入、关联的事实列表、重要性               | 是                      |
| **经验节点**   | ProceduralNode | 可复用的操作方法/互动策略            | 经验文本、向量嵌入、关联的子目标、回报分数、来源            | 是                      |
| **子目标节点**  | SubgoalNode    | 对经验进行索引的意图抽象             | 子目标文本、向量嵌入、关联的经验列表、是否激活、使用频率        | 是                      |

### 1.2 两条知识树

```
树1：事实知识（用户是谁 / 世界是什么）
  标签节点 ── 事实节点 ── 互动记录节点

树2：经验知识（怎么做 / 怎么互动）
  子目标节点 ── 经验节点 ── 互动记录节点
```

- **树1** 回答"是什么"类问题：检索入口为查询标签，通过标签投票 + 向量相似度的混合机制召回相关事实。
- **树2** 回答"怎么做"类问题：检索入口为预测子目标，通过子目标匹配召回相关经验。
- 两条树共享底层的互动记录节点，确保事实和经验可以追溯到同一段对话。

### 1.3 为什么是两条独立的树

事实检索和经验检索的查询输入完全不同：
- 查事实 → 输入是标签词（如"Python"、"偏好"、"GitHub Actions"）
- 查经验 → 输入是子目标描述（如"帮忙配置 CI/CD"）

分树设计让两种查询各自使用最优的检索策略，避免了用单一查询方式检索异质内容导致的精度损失。

### 1.4 节点数据结构详细设计（图数据库原生模型）

> **设计原则**：以下设计面向图数据库（如 Neo4j / NebulaGraph / JanusGraph）。节点仅包含**自身属性字段**，节点间关系通过 **边（Edge）** 对象独立存储。

各节点类名沿用 PlugMem 的核心命名：`EpisodicNode`、`SemanticNode`、`TagNode`、`ProceduralNode`、`SubgoalNode`。带 `（扩展）` 标注的字段为 Genban 在 PlugMem 基础上新增。

---

#### 1.4.1 EpisodicNode（互动记录节点）

每个 `EpisodicNode` 对应 Agent 交互流中的一步。同时也是被上层节点引用的证据来源。

| 字段名             | 字段类型            | 字段含义                           | 字段备注                              |
| :-------------- | :-------------- | :----------------------------- | :-------------------------------- |
| `episodic_id`   | `int`           | 节点主键，全局自增唯一                    | PlugMem 原生                        |
| `observation`   | `str`           | 该步观察到的内容（用户说了什么 / 系统返回了什么）     | PlugMem 原生                        |
| `action`        | `str`           | Agent 在该步执行的操作                 | PlugMem 原生                        |
| `subgoal`       | `str`           | 该步意图的自然语言概括                    | 由步骤 1.2 的 LLM 提取                  |
| `state`         | `str`           | 执行前环境/对话状态描述（由上一步的步骤 1.4 累积生成） | PlugMem 原生                        |
| `reward`        | `str`           | 该步效果的评估文本                      | 由步骤 1.3 的 LLM 生成                  |
| `time`          | `int`           | 该步发生的时间戳                       | PlugMem 原生                        |
| `session_id`    | `Optional[str]` | 所属会话标识，`None` 表示无会话归属          | PlugMem 原生                        |
| `user_id`       | `str`           | 用户id                           | Genban 扩展                         |
| `embedding`     | `List[float]`   | observation 文本的向量表示            | Genban 扩展。用于情景检索，避免 LLM 抽象过程的信息丢失 |
| `begin_chat_id` | `str`           | 起始消息id，用于定位原始交互内容              | Genban 扩展                         |
| `end_chat_id`   | `str`           | 结尾消息id，用于定位原始交互内容              | Genban 扩展                         |

**检索索引**：
- embedding：向量索引
- time：时间索引
- user_id：用户索引

---

#### 1.4.2 SemanticNode（事实节点）

从交互中提取的事实知识，是记忆系统最核心的资产。参与向量检索。

| 字段名                   | 字段类型            | 字段含义                               | 字段备注                 |
| :-------------------- | :-------------- | :--------------------------------- | :------------------- |
| `semantic_id`         | `int`           | 节点主键，全局自增唯一                        | PlugMem 原生           |
| `semantic_memory_str` | `str`           | 事实的自然语言陈述，如"用户偏好使用 Python 进行后端开发"  | PlugMem 原生           |
| `embedding`           | `List[float]`   | 事实文本的向量表示，维度取决于嵌入模型                | PlugMem 原生           |
| `time`                | `int`           | 逻辑时钟值，用于计算新近度（最近一次在交互中被重新提及/印证的时间） | PlugMem 原生           |
| `is_active`           | `bool`          | 是否活跃。`False` 表示该事实已被新信息覆盖或作废       | PlugMem 原生，默认 `True` |
| `credibility`         | `int`           | 可信度（0~10）。来自多次对话确认的铁证可信度更高         | PlugMem 原生           |
| `updated`             | `bool`          | 在单次合并周期中，标记是否曾被更新过（合并后标记为 `True`）  | PlugMem 原生           |
| `session_id`          | `Optional[str]` | 来源会话标识，`None` 表示跨会话提取              | PlugMem 原生           |
| `type`                | `str`           | 事实类型，分为“自动提取的事实”和“被要求记住的事实”        | Genban 扩展            |
| `user_id`             | `str`           | 用户id                               | Genban 扩展            |
| `usage_count`         | `int`           | 被召回次数                              | Genban 扩展            |

**检索索引**：
- embedding：向量索引
- is_active：是否生效索引
- user_id：用户索引

---

#### 1.4.3 TagNode（标签节点）

对事实进行分类的语义标签，是树1（事实知识树）的检索入口。

| 字段名            | 字段类型          | 字段含义                                        | 字段备注                                        |
| :------------- | :------------ | :------------------------------------------ | :------------------------------------------ |
| `tag_id`       | `int`         | 节点主键，全局自增唯一                                 | PlugMem 原生                                  |
| `tag`          | `str`         | 标签文本，如 `"Python"`、`"偏好"`、`"GitHub Actions"` | PlugMem 原生                                  |
| `embedding`    | `List[float]` | 标签文本的向量表示                                   | PlugMem 原生                                  |
| `importance`   | `float`       | 标签重要性分数，与引用次数和使用频率正相关                       | PlugMem 原生为 `int`（默认 1），Genban 建议改为 `float` |
| `time`         | `int`         | 逻辑时钟值，记录创建或最近更新时间                           | PlugMem 原生                                  |
| `user_id`      | `str`         | 用户id                                        | Genban 扩展                                   |
| `usage_count`  | `int`         | 被召回次数                                       | Genban 扩展                                   |
| `search_count` | `int`         | 被查询次数                                       | Genban 扩展                                   |

**重要性计算逻辑**：

```
importance = 引用次数 × 基础权重 + Σ(被关联 SemanticNode 的 Credibility) / 引用次数
             + 检索命中频率 × 命中权重
```

三种引用的含义：
- 该标签被多少条 `SemanticNode` 引用（通过 `TAGGED_AS` 边的入度计算）
- 该标签在检索中作为查询条件的频率（`usage_count`）
- 该标签被召回后在 LLM 推理中被实际引用的频率（`search_count`）

**标签唯一性保证**：通过 `tag` 文本精确匹配 + `embedding` 余弦相似度（>0.9 视为同义，触发合并判断）来保证不重复。

**检索索引**：
- embedding：向量索引
- user_id：用户索引

---

#### 1.4.4 ProceduralNode（经验节点）

可复用的操作方法或互动策略，是树2（经验知识树）的核心内容。

| 字段名                     | 字段类型            | 字段含义                                             | 字段备注                |
| :---------------------- | :-------------- | :----------------------------------------------- | :------------------ |
| `procedural_id`         | `int`           | 节点主键，全局自增唯一                                      | PlugMem 原生          |
| `procedural_memory_str` | `str`           | 经验的自然语言描述，如"配置 GitHub Actions 时需注意用户的 Node 版本偏好" | PlugMem 原生          |
| `embedding`             | `List[float]`   | 经验文本的向量表示                                        | PlugMem 原生          |
| `time`                  | `int`           | 逻辑时钟值，用于计算新近度                                    | PlugMem 原生          |
| `return`                | `int`           | 回报分数（0~10），该经验的历史效果评分                            | PlugMem 原生，来自步骤 2.4 |
| `session_id`            | `Optional[str]` | 来源会话标识                                           | PlugMem 原生          |
| `type`                  | `str`           | 经验类型，分为“自动提取的经验”和“被要求记住的经验”                      | Genban 扩展           |
| `user_id`               | `str`           | 用户id                                             | Genban 扩展           |
| `usage_count`           | `int`           | 在检索中被命中并实际使用的次数                                  | Genban 扩展           |

**`Return` 的生命周期**：
- 初始值：由 LLM 在步骤 2.4 中评估（可选，默认为 0.0）
- 更新时机：每次该经验被召回并在实际交互中产生效果后，根据用户反馈调整
- 更新方式：指数移动平均（EMA，`new_return = α × feedback + (1-α) × old_return`）

**检索索引**：
- embedding：向量索引
- user_id：用户索引

---

#### 1.4.5 SubgoalNode（子目标节点）

对经验进行索引的意图抽象，是树2（经验知识树）的检索入口。

| 字段名            | 字段类型          | 字段含义                          | 字段备注       |
| :------------- | :------------ | :---------------------------- | :--------- |
| `subgoal_id`   | `int`         | 节点主键，全局自增唯一                   | PlugMem 原生 |
| `subgoal`      | `str`         | 子目标的自然语言陈述，如"帮用户配置 CI/CD 流水线" | PlugMem 原生 |
| `embedding`    | `List[float]` | 子目标文本的向量表示                    | PlugMem 原生 |
| `time`         | `int`         | 逻辑时钟值                         | PlugMem 原生 |
| `user_id`      | `str`         | 用户id                          | Genban 扩展  |
| `usage_count`  | `int`         | 在检索中被命中并实际使用的次数               | Genban 扩展  |
| `search_count` | `int`         | 被查询次数                         | Genban 扩展  |

**检索索引**：
- embedding：向量索引
- user_id：用户索引

#### 1.4.6 边类型定义（关系数据对象）

以下边类型覆盖了记忆图中所有的节点关系。在图数据库中，每条边是一个独立的数据对象，包含源节点 ID、目标节点 ID、边类型标签和可选的边属性。

| 边类型              | 方向                                | 基数  | 含义         | 边属性       |
| :--------------- | :-------------------------------- | :-- | :--------- | :-------- |
| `EXTRACTED_FROM` | `SemanticNode` → `EpisodicNode`   | N:N | 事实从互动记录中提取 | `user_id` |
| `DERIVED_FROM`   | `ProceduralNode` → `EpisodicNode` | N:N | 经验从互动记录中总结 | `user_id` |
| `TAGGED_AS`      | `SemanticNode` ↔ `TagNode`        | N:N | 事实被打上标签    | `user_id` |
| `SERVES`         | `ProceduralNode` ↔ `SubgoalNode`  | N:N | 经验服务于子目标   | `user_id` |
| `NEXT_STEP`      | `EpisodicNode` → `EpisodicNode`   | 1:1 | 同轨迹内时序后继   | `user_id` |

**反向遍历**：图数据库天然支持沿边反向查询。

**检索索引**：
- user_id：用户索引

## 二、记忆写出流程（异步旁路）

记忆系统作为独立线程运行，持续监听 Agent 与用户/系统的交互流。整个写出流程分为三个阶段。

### 阶段一：交互流缓冲与轨迹切割

**目标**：将连续的交互流切分为有意义的小段（轨迹），每条轨迹围绕一个统一的意图。

> **关键机制**：步骤 1.2 ~ 1.4 并非每步独立执行，而是通过 **`state` 的滚动传递**形成跨步依赖——步骤 1.4 产出的新 `state` 成为下一步步骤 1.2/1.3/1.4 的输入。

每一步的处理顺序为：提取子目标 → 评估奖励 → 检测子目标切换 → 更新状态。其中步骤 1.2 使用总目标作为目标参数，结合上一步累积的状态和当前观察、动作来推断子目标；步骤 1.3 以刚提取的子目标（而非总目标）作为目标，评估该动作是否有效推进了这个子目标；步骤 1.4 重新回到总目标视角，将动作和观察整合进追踪状态，产出的新状态供下一步使用。第一步的初始状态为空字符串。

---

#### 步骤 1.1：获取交互步

从 Agent 核心获取最新的一步交互信息：

- 输入：交互事件流
- 输出：一步交互记录（用户输入 / Agent 动作 / 系统返回 / 时间戳 / 对话轮次）

#### 步骤 1.2：提取子目标

调用 LLM，根据总目标、上一步累积的状态、当前步的观察内容和 Agent 执行的动作，推断这一步的子目标——即一个自然语言短句，描述当前动作背后的直接意图。

- 输入：总目标 + 上一步累积状态 + 当前观察 + 当前动作
- 输出：子目标描述（自然语言）
- LLM 调用：是

> **Prompt 内容**：
>
> **系统指令**：你正在分析一个智能 Agent（基于大语言模型的 Agent）在动态环境中朝向特定目标执行的行为。
>
> **用户指令**：
> ```
> 在时刻 t，该 Agent 基于其状态、观察和总目标执行了一个动作。
> 你的任务是推断子目标——能够最好地解释 Agent 为何选择该动作的直接或中间目标。
>
> 请使用以下信息作为上下文：
> 总目标：{goal}
> 当前状态（过去上下文的摘要）：{state}
> 当前观察：{observation}
> 时刻 t 的动作：{action}
>
> 步骤 1：推理
> 分析当前状态和观察如何与总目标关联。
> 解释该动作如何帮助 Agent 推进目标——可能是通过达成某个较小的中间目标。
> 请做到明确且有因果性：描述为什么在这个上下文中该动作是有意义的。
>
> 步骤 2：子目标推断
> 在推理之后，推断 Agent 可能的子目标——一句简短的自然语言陈述，描述该动作背后的直接目的。
>
> 输出格式：
> ### 推理
> [你的推理过程——几句话解释该动作如何与目标和上下文关联]
> ### 子目标
> [一句短句描述推断出的子目标]
> ```

> 第一步的状态为空字符串。

#### 步骤 1.3：评估奖励

调用 LLM，以步骤 1.2 刚提取的子目标为目标，结合上一步状态、当前动作以及动作执行后的观察，评估该动作的实际效果。

- 输入：刚提取的子目标 + 上一步累积状态 + 当前动作 + 动作执行后的观察
- 输出：奖励评估文本（自然语言描述该动作是否有效推进了子目标）
- LLM 调用：是

> **Prompt 内容**：
>
> **系统指令**：你是一位专家级的轨迹评估者。你的任务是分析 Agent 在一个更大的目标导向任务中的一步决策过程。
>
> **用户指令**：
> ```
> 你将获得以下信息：
> 目标：Agent 的总体目标。
> 状态（时刻 t）：Agent 在执行动作之前已知和已完成的信息。
> 动作（时刻 t）：Agent 选择的单个动作。
> 观察（时刻 t+1）：该动作产生的即时结果。
>
> 你的任务是推断奖励——即用自然语言评估 Agent 的动作如何对实现目标产生贡献（正面或负面），基于产生的观察结果。
>
> 请仔细遵循以下步骤：
> 1. 推理过程：
> 基于当前状态解释该动作与目标之间的关系，以及观察结果是否符合预期的有利或不利结果。
> 考虑该动作是否推进了进展、造成了倒退、揭示了有用的新信息，还是白费力气。
> 总结你对该动作对目标的因果贡献的推理。
>
> 2. 最终奖励：
> 使用描述性语言，写出一句简洁的自然语言评估，评估 Agent 的动作。
> 奖励应该表达该动作在多大程度上以及以何种方式帮助或阻碍了目标的达成。
>
> 输入：
> 目标：{goal}
> 状态（时刻 t）：{state}
> 动作（时刻 t）：{action}
> 观察（时刻 t+1）：{observation}
>
> 输出格式：
> ### 推理
> [你的推理过程]
> ### 奖励
> [评估 Agent 动作的自然语言奖励陈述]
> ```

> **注意**：此步骤的目标参数是刚提取的 **子目标**，不是总体任务目标。奖励评估的语义是"该动作是否有效推进了子目标"，而非"是否推进了总目标"。

#### 步骤 1.4：更新状态

调用 LLM，以总目标为视角，将当前动作和动作执行后的观察整合进追踪状态，生成一个自然语言摘要，描述截至当前步的上下文、进度和关键信息。产出的新状态将作为下一步步骤 1.2/1.3/1.4 的输入。

- 输入：总目标 + 执行前状态 + 当前动作 + 动作执行后的观察
- 输出：更新后的状态描述
- LLM 调用：是

> **Prompt 内容**：
>
> **系统指令**：你是一个大语言模型推理引擎，负责基于 Agent 的持续轨迹更新其内部状态表示。
>
> **用户指令**：
> ```
> 你将获得四部分信息：
> 目标：Agent 当前的目标或任务。
> 前一状态（时刻 t）：一段自然语言摘要，描述 Agent 的上下文、历史以及截至目前的部分进展。
> 动作（时刻 t）：Agent 决定执行的下一步动作，以自然语言表达。
> 观察（时刻 t+1）：该动作产生的结果或反馈。
>
> 你的任务是推导出更新后的新状态——一段连贯的自然语言摘要，将前一状态、动作和新观察中的所有相关信息整合起来。
>
> 应遵循的步骤：
> 解读输入：检查目标、前一状态、动作和观察，理解 Agent 的情况发生了什么变化，包括位置和时间的详细信息。
> 推理更新：描述新状态与前一状态之间应有差异的逻辑过程。识别取得了什么进展、获得了什么新信息，以及上下文或关注点可能如何转移。
> 生成更新后的状态：写出一段清晰简洁的自然语言描述，总结 Agent 在时刻 t+1 的新状态。新状态应：
> - 包含动作和观察中的所有详细信息，特别是位置和时间的信息
> - 整合最新动作和观察的结果
>
> 输出格式：
> ### 推理
> （逐步解释新状态应如何基于输入进行更新。）
> ### 状态
> （在此提供最终的更新后状态摘要。）
>
> ---
> 输入：
> 目标：{goal}
> 前一状态（时刻 t）：{state}
> 动作：{action}
> 观察（时刻 t+1）：{observation}
> ```

#### 步骤 1.5：检测子目标切换

将当前步的子目标向量化，与上一步的子目标向量计算余弦相似度。如果相似度低于阈值（参考值 0.75），说明意图发生了显著变化——将之前缓冲的步骤打包为一条轨迹，清空缓冲区，开始新轨迹。

- 输入：相邻两步的子目标向量
- 输出：切割后的轨迹段（或继续缓冲）
- LLM 调用：否（纯向量计算）

> **设计要点**：切割阈值控制轨迹的"纯度"。阈值越低，轨迹越粗（一条轨迹可能包含多个弱关联步骤）；阈值越高，轨迹越细（切换频繁，可能切断相关步骤）。建议从 0.75 开始，根据实际效果调优。

---

### 阶段二：从轨迹中提取结构化知识

**目标**：当一条轨迹切割完成或被主动关闭时，从轨迹中提取事实和经验。

#### 步骤 2.1：从步骤中提取事实

遍历轨迹中的每一步，对每一步的观察内容调用 LLM，提取其中包含的事实。每条事实包含一项自然语言陈述和若干分类标签。

- 输入：每一步的观察文本
- 输出：事实列表，每项包含 { 陈述文本, 标签列表, 来源 episodic_id }
- LLM 调用：是（每个步骤 1 次）

> **Prompt 内容**：
>
> **系统指令**：你是一位专家级的信息提取者。
>
> **用户指令**：
> ```
> 你是一位从文档中提取精确、事实性信息的专家。
> 你的输出必须优先考虑具体性、避免歧义、消除冗余，并严格遵循以下所有格式规则。
>
> **核心指令：**
>
> 1) 事实/陈述提取与去重：
>    * 从文档中识别出互不相同的、事实性的陈述。
>    * 解决模糊引用：如果陈述的主语是代词或模糊描述（如"这部电影"、"那个乐队"、"那家公司"、"他"、"她"、"他们"），你必须基于对整篇文档的理解重写该陈述，使主语成为从文档中提取的完全明确的、具体的实体名称。不得在最终的事实/陈述中保留模糊的主语。
>    * 模糊引用解决示例：
>      错误：「这部电影由 xxx 执导。」「这部电影由 xxx 制作。」
>      正确：「Vaada Poda Nanbargal 由 Manikai 执导。」「Vaada Poda Nanbargal 由 xxx 制作。」
>    * 具体措辞：每一条陈述必须使用明确的、可识别的名称或标题。禁止使用任何模糊引用，包括但不限于："那次巡演"、"那部电影"、"那个乐队"、"它"、"他"、"她"或"他们"。
>    * 陈述长度策略（重要）：每条事实/陈述不必是单个短句。一条陈述可以是一个紧凑的多句段落，将紧密相关的信息归为一组，但总共最多包含 4 句话。这些句子应来源于原始文档材料，仅在解决模糊引用时做轻微编辑。
>    * 避免冗余：你必须将相似或重叠的事实合并为单一的、综合的陈述。不要创建多条重复相同核心信息、只有微小变化的陈述。
>
> 2) 标签生成：
>    * 为每条事实/陈述生成一个标签列表。
>    * 每条事实的标签数量是灵活的，应反映该陈述的信息密度。
>    * 标签应涵盖关键信息片段，例如：实体名称、年份、数字、国籍、语言、类型、角色、对象类型、描述性词语等。
>    * 大部分标签应为从陈述中逐字拷贝的短文本片段（精确子串）。这些标签直接源于文本的表层形式。
>    * 在以下情况下，你可以偶尔创建不属于当前陈述字面子串的额外标签：
>        - 你将相邻词语合并为一个短语（如"浪漫喜剧电影"）。
>        - 你从文档的另一部分导入一个表层片段（精确短语），以使当前陈述的含义明确。
>    * 当一个陈述的主语是代词或模糊描述时，你必须至少添加一个标签，用文档其他地方出现的精确表层形式明确命名该底层实体。
>    * 如果标签是动词，你必须使用其词干形式（如 `play`，而非 `played`、`playing` 或 `to play`）。
>    * 禁止使用元标签或本体式类别名：你不得使用诸如"名称"、"人物"、"年份"、"日期"、"城市"、"国家"、"地点"、"类型"、"语言"、"电影标题"等元标签。标签不是类型标签，而是承载内容的短语。
>    * 标签应相对简短。形容词和修饰语如"印度的"、"泰米尔语的"、"浪漫的"、"喜剧的"，如果出现在陈述中，通常应分别作为独立的标签。
>
> **输出约束：**
>    * 最多提取 10 条事实，但优先考虑质量而非数量。如果真正独立的事实少于 10 条，就输出更少。
>    * 每条事实必须提供其他事实没有覆盖的独特信息。
>    * 绝对不允许有任何泛泛的引用——每条陈述必须明确命名具体实体。
>
> **文档：**
> {observation}
>
> **输出格式：**
> ### 事实
> 1. **陈述：** [陈述内容]
>    **标签：** [标签0, 标签1, 标签2, 标签3, ...]
> 2. **陈述：** [陈述内容]
>    **标签：** [标签0, 标签1, 标签2, 标签3, ...]
> ...
> ```

> **注意**：事实提取的粒度是"每个步骤"，不是"每条轨迹"。一个 5 步的轨迹会产出 5 组事实，每组可能包含多条陈述。

#### 步骤 2.2：从轨迹中提取经验

将整条轨迹的步骤序列拼接为文本（每步包含：状态、动作、奖励），调用 LLM 总结过程中的可复用经验。

- 输入：轨迹文本（步骤序列）
- 输出：经验描述 + 概括子目标
- LLM 调用：是（每条轨迹 1 次）

> **Prompt 内容**：
>
> **系统指令**：你是一个分析模型，经过训练用于通过记录的轨迹来总结智能 Agent 的行为。
>
> **用户指令**：
> ```
> 你将获得一段 Agent 的轨迹记录。
> 你的任务是分析该轨迹并推导出：
> - Agent 正在追求的一个主要目标。
> - 一条经验洞察——一段简洁的反思，总结 Agent 的行为。
>
> 生成回复时请遵循以下过程：
> 1. 推理：
> 分析轨迹记录，写下可概括的信息和模式，这些作为记忆将对未来的任务有用。
>
> 2. 输出目标和经验洞察：
> 用一句抽象的语言描述该轨迹的总体目标。
> 用一段自然语言清晰表达经验洞察和总结 Agent 行为的反思。
>
> 输出格式：
> ### 推理
> [你的推理过程]
> ### 目标
> [一句总结目标的话]
> ### 经验洞察
> [表达经验洞察的段落]
>
> ---
> 输入：
> 轨迹：{trajectory}
> ```

#### 步骤 2.3：生成向量嵌入

对以下内容调用嵌入模型生成向量：

- 每个事实的陈述文本
- 每个标签的文本
- 每个子目标的文本
- 每条经验的经验文本

- 输入：文本
- 输出：向量表示（浮点数数组）
- LLM 调用：否（使用嵌入模型）

#### 步骤 2.4：评估经验回报（可选）

调用 LLM 对步骤 2.2 提取的经验进行价值评分。

- 输入：子目标 + 经验描述
- 输出：回报分数（如 0.0~10.0）
- LLM 调用：是（可选，每条轨迹 1 次）

> **Prompt 内容**：
>
> **系统指令**：你是一位评估者。你的任务是判断 Agent 对给定目标的追求和完成程度。
>
> **用户指令**：
> ```
> 你将获得：
> 目标描述：一段文字，解释 Agent 试图达成什么。
> 过程描述：一段文字，描述 Agent 的动作、决策以及向该目标推进的进展。
>
> 你的任务：
> 分析 Agent 的过程，判断目标在多大程度上被完成，考虑以下方面：
>
> 评分标准（分数 1-10）：
> 10：Agent 完全完成了目标，没有重大遗漏；动作与目标完全一致。
> 8-9：Agent 完成了目标的绝大部分，仅有微小的缺口；一致性很强但不完美。
> 6-7：部分完成；Agent 覆盖了许多关键要素，但留下了明显的未完成或执行不力的部分。
> 4-5：进展有限；Agent 尝试了该目标，但完成了不到一半，或以低效的方式完成。
> 2-3：几乎无进展；动作与目标几乎不相关，或仅取得了极少的结果。
> 1：无意义进展；动作对达成目标完全无贡献。
>
> 重要说明：
> 仅基于完成程度和与所述目标的一致性评分。除非特别要求，否则不要提供解释或评论。
> 输出必须严格遵循以下格式。
>
> 输出格式：
> ### 分数
> [1 到 10 的数字]
>
> 输入：
> ---
> 目标：
> {subgoal}
> 过程：
> {procedural_memory}
> ```

### 阶段三：将结构化知识写入记忆图

**目标**：将提取出的事实、经验、标签、子目标写入图结构，并处理新旧知识的合并。

#### 步骤 3.1：写入互动记录

将轨迹中的每一步作为一个互动记录节点写入存储。

- 向量计算：对每个 EpisodicNode 的 `observation` 文本生成 `embedding`
- 记录字段：观察内容、执行动作、子目标、状态、奖励、时间戳、会话ID等
- 创建关系：在同一轨迹内，按步骤顺序创建 `NEXT_STEP` 边（`EpisodicNode[n] → EpisodicNode[n+1]`），轨迹最后一步不连出边

#### 步骤 3.2：创建事实节点

对每条提取出的事实创建一个事实节点。

- 向量计算：对事实文本生成 `embedding`
- 记录字段：事实文本、向量嵌入等
- 创建关系：创建 `EXTRACTED_FROM` 边来关联互动记录

#### 步骤 3.3：查找/创建标签节点

对事实节点的每个标签，检查是否已存在同名标签节点：

- 如果存在：将该事实节点添加到已有标签节点的关联列表中
- 如果不存在：创建新的标签节点，对标签文本生成 `embedding` ，记录标签文本和向量嵌入等

#### 步骤 3.4：建立标签节点关联并更新标签重要性

对步骤 3.3 中的每个标签，完成关联和重要性更新

- 创建关系：建立 `TAGGED_AS` 边来关联事实节点
- 更新标签重要性：对每个标签节点，更新其重要性参数

#### 步骤 3.5：事实合并与去重

对每条新的事实节点，在已有的事实库中检索语义最接近的已有事实。

- 使用向量相似度做初步筛选（取 Top-N 相似事实）
- 如果相似度超过合并阈值（如 0.9），调用 LLM 判断两条事实的关系：
  - **同一事实的更新**：新信息覆盖旧信息，旧事实标记为不活跃
  - **可合并的相似事实**：合并为一条更完善的表述
  - **弱相关但不相同**：两者都保留
- 根据 LLM 判断结果决定是合并、更新还是新增
- LLM 调用：是（每个需要合并的候选对 1 次）

> **Prompt 内容**：
>
> **系统指令**：你的任务是将两条相似的信息合并为一条连贯的陈述。
>
> **用户指令**：
> ```
> 你获得了关于同一话题的两条记忆项。其中一条更早（信息1），另一条更晚（信息2）。
>
> 你的任务：
> (1) 将它们合并为一条改进后的、清晰、简洁的陈述。不要编造新事实。
> (2) 决定在合并后是否停用（软删除）原来的两条节点。
>
> 输入：
> 信息1（较早的信息）：{memory_earlier}
> 信息2（较晚的信息）：{memory_later}
>
> 停用决策规则（恰好选择一种情况）：
>
> 情况A：「更新同一事实」
> - 条件：信息1和信息2本质上描述的是同一事实/事件，信息2主要更新/修正/细化了信息1的细节。
> - 处理：停用两条原始信息，因为合并后的节点完全取代了它们。
>
> 情况B：「同一主题可合并」
> - 条件：信息1和信息2在同一主题下强相关，且合并后的陈述读起来像统一的摘要（而不是生硬的拼接）。
> - 处理：停用两条原始信息。
>
> 情况C：「弱相关有拼接风险」
> - 条件：信息1和信息2仅弱相关；合并感觉像是在生硬拼接两段内容；停用任何一条原始信息都可能损害未来的检索，因为合并后的嵌入可能变得语义模糊。
> - 处理：两条原始信息都不停用。
>
> 硬性约束：
> - 输出必须是合法的 JSON（不允许 Markdown 和额外文本）。
> - relationship 必须恰好是以上三种标签之一。
> - 如果是情况A或B => deactivate_earlier=true 且 deactivate_later=true。
> - 如果是情况C => deactivate_earlier=false 且 deactivate_later=false。
> - 如果两条记忆冲突，以信息2（更晚的信息）为准。
> - 输出你做此决定的简要推理。
>
> 输出必须是合法的 JSON，恰好包含以下字段：
> - merged_statement（字符串）
> - relationship（"UPDATE_SAME_FACT" | "SAME_TOPIC_MERGE_WELL" | "WEAK_RELATED_STITCH_RISK"）
> - deactivate_earlier（布尔值）
> - deactivate_later（布尔值）
> - simple_reasoning（字符串）
>
> 仅返回 JSON 对象。
> ```

#### 步骤 3.6：创建经验节点

对每条提取出的经验创建一个经验节点。

- 记录字段：经验文本、向量嵌入、回报分数
- 关联到对应的互动记录节点

#### 步骤 3.7：创建/合并子目标节点

对经验节点的子目标，检查是否已存在同义子目标节点：

- 如果存在：调用 LLM 将新旧子目标合并为一个更完善的表述，更新已有子目标节点的向量嵌入
- 如果不存在：创建新的子目标节点
- LLM 调用：是（需要合并时）

> **Prompt 内容**：
>
> **系统指令**：你是一个将两个相似目标合并为一个统一目标的助手。
>
> **用户指令**：
> ```
> 每个目标可能包含重叠或互补的信息。
> 你的任务是仔细地将它们合并为一个单一的、连贯的、结构良好的目标，保留两者的所有重要细节。
> 避免冗余，确保合并后的目标听起来自然且语气一致。
>
> 输入：
> 较早的目标：{goal_1}
> 较晚的目标：{goal_2}
>
> 输出：
> 合并后的目标：[在此写出统一的目标]
> ```

#### 步骤 3.8：建立经验与子目标关联

将经验节点挂载到对应的子目标节点下，建立双向关联（`SERVES` 边，`ProceduralNode ↔ SubgoalNode`）。子目标节点是否可用由其 `SERVES` 边出度是否 > 0 判断。

## 三、记忆召回流程（同步，在回复生成前）

当用户输入到达时，记忆系统在 Agent 生成回复之前完成以下步骤。整个召回流程分为五个阶段。

### 阶段一：检索规划

**目标**：理解用户当前输入需要什么类型的记忆，并生成检索用的查询条件。

#### 步骤 R1.1：生成查询条件

调用 LLM，根据当前上下文同时输出两样东西：

- (a) 预测的下一子目标：Agent 接下来应该达成什么
- (b) 查询标签列表：一组用于检索事实的标签词

- 输入：总目标 + 当前子目标 + 当前状态 + 用户输入
- 输出：{ 预测子目标, 查询标签列表 }
- LLM 调用：是

> **Prompt 内容**：
>
> **系统指令**：你是一个助手，其职责是为一个 LLM Agent 决定应该召回什么记忆。
>
> **用户指令**：
> ```
> 你是一位专家，负责分析 Agent 的目标和当前观察，为一个目标导向的问答系统生成检索标签。
>
> 你的设定：
> - 目标：Agent 要完成的总体目标。
> - 当前子目标：Agent 当前正在追求的子目标（可能为空）。
> - 当前状态：对 Agent 当前内部状态的描述（可能为空）。
> - 输入（当前观察）：Agent 最近的观察内容（通常是一个问题或任务指令）。
> - 任务：提取一组优先级排序的高质量标签，这些标签最可能检索到直接有助于完成目标的信息。
>
> 指令：
> 1) 目标导向的标签选择（关键）：
> - 仔细阅读目标和当前观察。
> - 仅生成那些极有可能检索到解决/完成目标所需证据的标签。
> - 优先选择能够标识以下内容的标签：
>   - 目标所询问的目标实体（人物、组织、地点、作品、事件）。
>   - 由观察暗示的可能需要用于多跳检索的桥接实体。
>   - 明确的约束条件：日期、年份、角色、头衔、唯一描述词、数字。
> - 避免低信号或泛泛的标签，它们不太可能检索到有用的证据。
>
> 2) 具体的、有依据的标签：
> - 大部分标签必须是从目标或当前观察中逐字拷贝的短文本片段（精确子串）。
> - 你可以添加少量非字面标签，但仅当它们是简短的、被强烈暗示的，且对检索明显必要时（如标准名称扩展或常见别名）。
> - 如果标签是动词，必须使用其词干形式。
>
> 3) 优先级和数量：
> - 总共输出 5-12 个标签（不是"尽可能多"）。
> - 按预期检索有用程度排序（最有用的排在前面）。
> - 确保标签承载内容且相对简短。
>
> 4) 关键——禁止的标签：
> - 不要生成标签 "user"。
> - 不要使用元标签或类型名，如"名称"、"人物"、"年份"、"日期"、"城市"、"国家"、"地点"、"类型"、"电影标题"等。
>
> 输出格式：
> ### 推理
> [你分析信息并完成任务的过程]
> ### 标签
> **标签：** ["标签0", "标签1", "标签2", "标签3", ...]
> ### 下一子目标
> [Agent 接下来应该追求的单个最佳子目标。]
>
> ---
> 输入：
> 目标：{goal}
> 当前子目标：{subgoal}
> 当前状态：{state}
> 当前观察：{observation}
> ```

### 阶段二：事实路径检索

**目标**：从记忆图中召回与当前查询最相关的事实。采用"标签投票 + 向量相似度混合"的两阶段检索策略。

#### 步骤 R2.1：向量相似度初筛

将用户输入的文本向量化，与所有活跃的事实节点计算余弦相似度，取相似度最高的 5 个作为"种子候选"。

- LLM 调用：否

#### 步骤 R2.2：标签匹配与扩展

遍历查询标签列表中的每个标签词，在标签节点库中通过向量相似度检索语义匹配的标签节点（超过匹配阈值才保留）。收集每个匹配标签下挂载的所有事实节点。

- LLM 调用：否

#### 步骤 R2.3：建立投票计数

对所有候选事实节点建立投票计数：

- 标签精确匹配：该节点被某个查询标签的标签节点直接关联 → +5 分
- 间接标签关联：该节点被与查询标签向量相似度 > 0.8 的相邻 TagNode 关联 → +1 分
- 种子候选加成：该节点来自步骤 R2.1 的种子候选列表 → +2 分
- 归一化：将总重要性分数除以该事实节点关联的标签数量（`TAGGED_AS` 边出度）

#### 步骤 R2.4：计算综合价值分数

将所有候选事实节点（种子候选 + 标签召回）收集起来，计算每条事实的最终价值分数：

```
最终分数 = 向量相关性 × w1 + 标签投票重要性 × w2 + 事实可信度 × w3
```

- 权重 w1、w2、w3 可配置
- LLM 调用：否

#### 步骤 R2.5：排序与截断

按价值分数降序排列，取前 K 条（参考值 10）。过滤掉分数低于检索阈值的事实，得到最终召回的事实列表。

---

### 阶段三：经验路径检索

**目标**：从记忆图中召回与当前子目标最相关的可复用经验。

#### 步骤 R3.1：子目标匹配

将步骤 R1.1 预测的"下一子目标"向量化，在子目标节点库中检索语义最匹配的子目标节点。

- 使用向量相似度 + 阈值过滤
- 过滤条件：仅保留至少有一条出向 `SERVES` 边的 SubgoalNode（确保匹配到的子目标有可用经验）
- LLM 调用：否

#### 步骤 R3.2：收集关联经验

如果匹配到子目标节点，收集该子目标下挂载的所有经验节点。

#### 步骤 R3.3：计算经验价值分数

对每个候选经验节点计算价值分数：

```
最终分数 = 子目标相似度 × w1 + 经验回报值 × w2 + 经验新近度 × w3
```

- 新近度 = 当前时间 - 经验创建时间（越新越高）
- 回报值 = 历史评估的经验价值
- LLM 调用：否

#### 步骤 R3.4：排序与截断

按分数排序取前 K 条，过滤低于阈值的结果，得到最终召回的经验列表。

### 阶段四：情景路径检索（可选）

**目标**：当需要完整的对话上下文时，通过两路互补的方式定位相关会话。

**双路检索策略**：情景检索有两条互补的入口——

| 路径 | 检索方式 | 适用场景 |
|:---|:---|:---|
| **直接向量检索**（Genban 扩展） | 将用户输入向量化，直接在 EpisodicNode 的 embedding 向量集合中检索语义相似的 observation | 语义匹配模糊但原始文本保真度高，召回命中后沿 `NEXT_STEP` 边展开上下文 |
| **事实投票路由**（PlugMem 原有） | 通过事实节点的 session_id 投票定位相关会话 | 结构化路径，精确度高但受 LLM 抽象过程信息丢失影响 |

> **建议**：两条路径可并行执行，结果合并后按相关度分数统一排序。直接向量路径保底覆盖 LLM 抽象遗漏的信号，事实投票路径提供高质量的结构化匹配。

#### 步骤 R4.1：直接向量检索（Genban 扩展）

将用户输入向量化，在 EpisodicNode 向量集合中检索 Top-K 相似 observation。

- 命中后沿 `NEXT_STEP` 边正向/反向各展开 2~3 步，还原局部对话上下文
- LLM 调用：否

#### 步骤 R4.2：收集会话 ID（事实投票路由）

从步骤二的检索结果中，收集所有事实节点关联的互动记录节点的会话 ID（通过 `EXTRACTED_FROM` 边反向遍历）。

#### 步骤 R4.3：会话投票

对会话 ID 做投票计数：同一会话被超过 N 条事实引用的（参考值 ≥3），说明该会话与当前查询高度相关，召回该会话的完整对话记录。沿 `NEXT_STEP` 边遍历可高效还原会话的完整时序。

#### 步骤 R4.4：碎片化记忆回退

未被足够事实引用的会话，仅将其中的事实节点文本作为碎片化记忆返回。

### 阶段五：组装增强上下文

**目标**：利用 LLM 将多来源检索到的记忆材料进行智能筛选、融合与重组，输出一段可直接注入 Agent 推理过程的紧凑上下文。

与简单的手动文本拼接不同，此步骤让 LLM 完成以下智能处理：

- **相关性过滤**：筛掉与当前目标/用户输入无关或冗余的记忆条目
- **矛盾消解**：当多条记忆存在冲突时，根据时间新近度、来源可信度做出取舍
- **信息融合**：将分散的事实、经验、情景整合为连贯的自然语言叙述
- **按需组织**：围绕当前目标和子目标，以对当前推理最有用的顺序组织信息

#### 步骤 R5.1：调用记忆上下文组装 LLM

将阶段二（事实列表）、阶段三（经验列表）、阶段四（情景记录）的检索结果，连同当前上下文（总目标、当前子目标、当前状态、用户输入），一并送入 LLM，由 LLM 完成智能组装。

- 输入：总目标 + 当前子目标 + 当前状态 + 用户输入 + 事实列表 + 经验列表 + 情景记录
- 输出：一段组装后的紧凑记忆上下文文本
- LLM 调用：是

> **Prompt 内容**：
>
> **系统指令**：你是一个记忆上下文组装器，负责从多条搜索到的记忆材料中提取、筛选、融合出对当前推理最有帮助的信息。
>
> **用户指令**：
> ```
> 你是一个记忆上下文组装器。你的任务是将搜索到的记忆材料整合为一段紧凑、准确的上下文，帮助 Agent 更好地完成当前推理。
>
> **你将收到以下信息：**
>
> 1) 当前上下文：
>    - 总目标：Agent 正在完成的总体任务
>    - 当前子目标：Agent 当前正在追求的具体子目标（可能为空）
>    - 当前状态：Agent 的内部状态描述（可能为空）
>    - 用户输入：用户当前的问题或指令
>
> 2) 检索到的记忆材料（可能有多类，也可能某一类为空）：
>    - 相关事实：从长期记忆中搜索到的与当前查询相关的事实陈述
>    - 相关经验：从长期记忆中搜索到的可复用操作经验
>    - 相关情景：从历史对话中搜索到的语义相似的对话片段
>
> **你的任务步骤：**
>
> 步骤 1 — 相关性筛选：
>   - 逐条审视每条记忆材料，判断其与当前上下文（总目标、子目标、用户输入）的相关性。
>   - 与当前问题完全无关的记忆直接丢弃。
>   - 部分相关的记忆保留其中相关的部分。
>   - 仅提供背景但不直接回答问题的记忆，标记为"背景参考"。
>
> 步骤 2 — 矛盾处理：
>   - 如果多条记忆之间存在矛盾或冲突：
>     * 优先采信时间更新的记忆（更接近当前状态）。
>     * 优先采信可信度更高的记忆。
>     * 优先采信来源更具体、更直接相关的记忆（情景 > 经验 > 孤立事实）。
>   - 如果无法确定哪条更可信，在输出中同时保留并注明矛盾。
>
> 步骤 3 — 信息融合：
>   - 将表达同一主题但分散在不同记忆条目中的信息合并。
>   - 避免重复表述相同内容。
>   - 如果情景记录中的对话片段包含的事实与事实列表中的条目一致，合并为一条综合描述。
>
> 步骤 4 — 按需组织：
>   - 将筛选融合后的信息，按照对当前目标/子目标最有帮助的顺序排列。
>   - 直接回答当前问题的信息放在前面。
>   - 提供背景和上下文的信息放在中间。
>   - 可能有用但关系间接的参考信息放在最后。
>
> 步骤 5 — 上下文合成：
>   - 将组织好的信息整合为一段连贯的自然语言文本。
>   - 保持简洁，总长度控制在 500 字以内（除非记忆材料本身信息密度极高）。
>   - 使用平实的陈述语气，不要添加原始记忆中没有的信息（除非是解决模糊指代所必需的补全）。
>   - 严禁编造记忆中没有的内容。
>
> **输出格式：**
>
> ### 推理过程
> [简要说明你如何处理这些记忆材料：哪些保留了、哪些丢弃了、如何处理矛盾、如何组织信息]
>
> ### 组装后的记忆上下文
> [一段连贯的自然语言文本，可直接作为 Agent 推理的上下文注入。不包含任何元信息或格式标记，仅包含内容本身。]
>
> ---
> **当前上下文：**
> 总目标：{goal}
> 当前子目标：{subgoal}
> 当前状态：{state}
> 用户输入：{observation}
>
> **检索到的记忆：**
> 相关事实：
> {semantic_memory}
>
> 相关经验：
> {procedural_memory}
>
> 相关情景：
> {episodic_memory}
> ```

#### 步骤 R5.2：注入回复生成

将步骤 R5.1 LLM 输出的"组装后的记忆上下文"作为增强上下文，附带到 Agent 的回复生成请求中。Agent 核心 LLM 将基于以下信息生成回复：

- 系统指令
- 当前总体目标
- 当前状态
- 用户输入
- 组装后的记忆上下文（来自步骤 R5.1）

## 四、全流程概览

### 写出流程

```
交互流 → [1.2 子目标] → [1.3 奖励] → [1.4 状态] → [1.5 轨迹切割]
                                                        ↓
                                        [2.1 事实提取] ← 沿每步执行
                                        [2.2 经验提取] ← 沿每条轨迹执行
                                        [2.3 向量嵌入]
                                                        ↓
                    ┌───────────────────────────────────┐
                    │  3.1 互动记录 → 3.2 事实节点       │
                    │  3.3 标签节点 ↔ 3.4 标签重要性     │
                    │  3.5 事实合并去重                  │
                    │  3.6 经验节点 → 3.7 子目标节点     │
                    │  3.8 经验-子目标关联               │
                    └───────────────────────────────────┘

涉及 LLM 调用：阶段一 3 次/步（子目标、奖励、状态）
              + 阶段二 1 次/步（事实提取）= 4 次/步
              + 1~2 次/轨迹（经验提取、可选回报评分）
```

### 召回流程

```
用户输入 → [R1.1 检索规划] → {预测子目标, 查询标签}
                                   │
              ┌────────────────────┼────────────────────┐
              ▼                    ▼                    ▼
    [R2 事实路径检索]    [R3 经验路径检索]    [R4 情景路径检索]
    ├ R2.1 向量初筛      ├ R3.1 子目标匹配     ├ R4.1 直接向量检索
    ├ R2.2 标签匹配      ├ R3.2 收集经验       ├ R4.2 收集会话ID
    ├ R2.3 投票计数      ├ R3.3 价值评分       ├ R4.3 会话投票
    ├ R2.4 综合评分      └ R3.4 排序截断       └ R4.4 碎片回退
    └ R2.5 排序截断
              │                    │                    │
              └────────────────────┼────────────────────┘
                                   ▼
                         [R5.1 LLM 智能组装] → [R5.2 注入回复]

涉及 LLM 调用：1 次（检索规划）+ 1 次（记忆上下文组装）
其余均为向量检索 + 数值计算，不涉及 LLM
```

## 五、实现建议

### 5.1 开发顺序

建议按以下顺序逐步实现和测试：

1. **数据结构定义**：五种节点的数据结构、向量存储选型（如 ChromaDB / FAISS / Milvus）
2. **写出流程阶段一**：交互流缓冲与轨迹切割（这是系统的入口，验证 LLM 提取子目标/奖励/状态的效果）
3. **写出流程阶段二**：结构化知识提取（事实、经验、向量嵌入）
4. **写出流程阶段三**：写入记忆图（节点创建、关联建立、事实合并去重）
5. **召回流程阶段二+三**：事实检索和经验检索（这是检索的核心逻辑）
6. **召回流程阶段一**：检索规划（规划结果直接影响后续检索质量）
7. **召回流程阶段五**：上下文组装与注入
8. **写出流程阶段三步骤 3.5**：事实合并去重（需要上线运行积累一定数据量后再调优）

### 5.2 关键参数

| 参数 | 参考值 | 说明 |
|:---|:---|:---|
| 子目标切割相似度阈值 | 0.75 | 低于此值触发轨迹切割 |
| 事实向量初筛 Top-K | 5 | 种子候选数量 |
| 事实检索最终 Top-K | 10 | 最终召回事实数量 |
| 经验检索最终 Top-K | 3~5 | 最终召回经验数量 |
| 事实合并相似度阈值 | 0.9 | 触发 LLM 合并判断 |
| 标签匹配相似度阈值 | 0.8 | 标签检索的匹配门槛 |
| 会话投票阈值 | 3 | 召回完整会话所需的最小事实引用数 |

### 5.3 扩展点

以下是在基础框架上可以逐步扩展的方向：

- **用户状态感知**：在标签体系中增加用户情绪/场景的结构化标签，支持情感感知的检索
- **经验分层**：区分通用经验（适用所有用户）和个人化经验（适用特定用户），在检索时按场景加权
- **敏感信息保护**：对特定事实标记敏感级别，在召回时走额外的权限检查
